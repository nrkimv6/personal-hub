# 모니터링 시스템 API 서버 PRD

## 1. 개요
쿠팡 API 및 네이버 예약 모니터링 시스템의 API 서버로, 기존 모니터링 시스템을 마이그레이션하여 확장성과 관리 용이성을 개선합니다.

## 2. 기술 스택
- FastAPI: API 서버 프레임워크
- Sqlite
- Pydantic: 데이터 검증
- Playwright: 웹 페이지 모니터링

- 작업 스케줄링 관리
- 날짜 기반 우선순위 스케줄링 시스템

- 대기열 관리 시스템
  - 모니터링 대상별 대기열 관리
  - 동시 실행 모니터링 수 제한
  - 대기열 처리기 자동 시작/중지
  - 모니터링 종료 시 자동 정리
  - 에러 발생 시 자동 재시도
  - 대기열 상태 모니터링 및 로깅
- 날짜 기반 간격 계산 시스템
  - 목표 날짜까지 남은 일수에 따른 동적 간격 조정

  - 목표 날짜 7일 전: 5분 간격
  - 목표 날짜 7일 이상 남음: 15분 간격
  - 자정 이후 첫 요청 시 간격 재계산
  - 날짜 변경 감지 및 자동 간격 조정
- 탭 공유 및 관리 시스템
  - 모니터링 작업 완료 후 탭 즉시 반환
  - 여러 URL 간 탭 공유로 리소스 효율화
  - 탭 요청 대기열 및 타임아웃 처리
  - 탭 회전 시스템으로 메모리 누수 방지
  - 탭 풀 관리로 리소스 최적화

#### 3.2.3 쿠팡 API 모니터링 시스템
- 쿠팡 API 응답 모니터링
- 예약 가능 상태 감지
- 매진 상태 감지
- 에러 상태 감지 및 처리
- 응답 데이터 파싱 및 분석
- 상태 변화 알림

##### 3.2.3.1 실행 조건
- 모니터링 대상의 service_type이 ServiceType.COUPANG인 경우에만 실행
- is_active와 is_enabled가 모두 True인 경우에만 실행
- 유지보수 모드가 아닐 때만 실행
- 예약된 유지보수 시간이 아닐 때만 실행

##### 3.2.3.2 모니터링 대상
- URL이 "https://trip.coupang.com/tp/products/"로 시작하는 쿠팡 여행 상품 페이지
- URL에서 상품 ID와 날짜 정보를 추출할 수 있어야 함
- 날짜 정보는 URL의 쿼리 파라미터(startDateTime 또는 startDate) 또는 URL 경로에서 추출

##### 3.2.3.3 모니터링 주기 및 에러 처리
- settings.COUPANG_CHECK_INTERVAL 설정값에 따라 결정
- 에러 발생 시 지수 백오프(exponential backoff) 방식으로 대기 시간 조정
- 최대 3번까지 재시도
- 재시도 간격은 기본 10초에서 지수적으로 증가

##### 3.2.3.4 상태 변경 감지
- 상품의 판매 상태(saleStatus) 변경
- 재고 수량(stockCount) 변경
- 상태 변경 시 알림 발송
- 에러 발생 시 로그 기록 및 알림 발송

#### 3.2.4 네이버 예약 모니터링 시스템

- 에러 상태 감지 및 처리
- 페이지 콘텐츠 파싱 및 분석
- 상태 변화 알림

##### 3.2.4.1 실행 조건
- 모니터링 대상의 service_type이 ServiceType.NAVER인 경우에만 실행
- is_active와 is_enabled가 모두 True인 경우에만 실행
- 유지보수 모드가 아닐 때만 실행
- 예약된 유지보수 시간이 아닐 때만 실행

##### 3.2.4.2 모니터링 대상
- URL이 "https://booking.naver.com/"로 시작하는 네이버 예약 페이지
- URL에서 예약 정보와 날짜 정보를 추출할 수 있어야 함
- 날짜 정보는 URL의 쿼리 파라미터(startDateTime 또는 startDate)에서 추출

##### 3.2.4.3 모니터링 주기 및 에러 처리
- 기본 모니터링 간격은 30초
- 사용자 정의 간격 설정 가능
- 에러 발생 시 지수 백오프(exponential backoff) 방식으로 대기 시간 조정
- 최대 3번까지 재시도
- 재시도 간격은 기본 10초에서 지수적으로 증가

##### 3.2.4.4 상태 변경 감지

## 4. API 엔드포인트 계획

### 4.1 모니터링 대상 관리
