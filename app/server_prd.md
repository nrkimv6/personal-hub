# 모니터링 시스템 API 서버 PRD

## 1. 개요
웹 페이지 모니터링 시스템의 API 서버로, 기존 모니터링 시스템을 마이그레이션하여 확장성과 관리 용이성을 개선합니다.

## 2. 기술 스택
- FastAPI: API 서버 프레임워크
- Sqlite
- Pydantic: 데이터 검증
- Playwright: 웹 페이지 모니터링

## 8. 특별 기능 구현 상세

### 8.1 스케줄링 및 작업 관리
#### 8.1.1 날짜 기반 스케줄링 시스템
##### 우선순위 기반 모니터링
- URL의 날짜 파라미터를 기반으로 모니터링 간격을 자동 계산하는 기능
- 미래 날짜에 따른 우선순위 기반 모니터링:
  - 오늘/이미 지난 날짜: 2-7초 간격으로 빠른 모니터링
  - 내일: 3-10초 간격으로 모니터링
  - 1-7일 이내: 20-50초 간격으로 중간 주기 모니터링 
  - 7일 이상: 200-300초 간격으로 낮은 주기 모니터링

##### 간격 설정 및 관리
- 사용자 정의 간격 설정 지원:
  - 자동 계산된 간격 외에도 사용자가 직접 간격 설정 가능
  - 사용자 정의 간격 설정 시 자동 재계산 방지 기능
- 날짜나 URL 변경 시 간격 자동 재계산 (사용자 정의 아닌 경우)

##### 하이브리드 스케줄링 방식
- 각 모니터링 대상별 독립적인 비동기 작업 사용
- 다음 실행 시간(next_run_time) 기반 실행 계획
- CPU 부하 분산을 위한 짧은 슬립 대기 방식 사용
- 동적 간격 계산과 데이터베이스 저장 간격의 장점 결합

#### 8.1.4 서버 자동 시작 및 중복 실행 방지 시스템
##### 서버 시작 시 자동 모니터링
- 서버 시작 시 활성화된(is_active=True) 모니터링 대상 자동 시작
- 부팅이나 서버 재시작 후 수동 개입 없이 모니터링 자동 재개
- 활성 상태별 필터링을 통한 선택적 시작 지원
- 자동 시작 시 상세 로깅으로 시작 상태 추적
- 개별 대상별 시작 성공/실패 기록

##### 중복 실행 방지 메커니즘
- 대상 ID 기반 모니터링 태스크 추적 시스템
- 이미 실행 중인 모니터링 태스크 자동 감지 및 중복 시작 방지
- 실행 상태 확인을 통한 안전한 태스크 관리
  - 태스크 상태가 완료되지 않았는지(not done) 확인
  - 기존 태스크 유지 및 중복 태스크 생성 방지
- API 엔드포인트 호출 및 서버 시작 시 모두 중복 방지 적용
- 태스크 취소와 완료 상태 명확한 구분

### 8.2 콘텐츠 분석 및 알림
#### 8.2.1 콘텐츠 파싱 시스템
##### 정보 추출 및 파싱
- HTML 콘텐츠에서 유용한 정보 추출하여 알림에 포함
- parse_time_and_stock(): 버튼 텍스트에서 시간과 매수 정보 추출
- parse_page_info(): 페이지에서 제목, 시간, 매수 등 종합 정보 추출
- 다양한 형식의 웹페이지 대응 가능한 유연한 파싱 로직
- 버튼 텍스트에서 정확한 시간 및 매수 정보 파싱
- 추가 시간 정보 파싱으로 더 풍부한 알림 메시지

#### 8.2.2 알림 메시지 강화
##### 맞춤형 알림 콘텐츠
- 추출된 정보를 사용자 친화적인 형태로 알림 메시지에 통합
- 페이지 정보 파싱을 통한 맞춤형 알림 콘텐츠:
  - 제목, 예약 가능 시간, 매수 정보 등
- 모니터링 경과 시간 정보 추가로 시스템 성능 확인
- 날짜 정보 추출 및 알림 메시지 포함
- 시/분/초 단위 가독성 높은 경과 시간 표시
- 유효성 검사 결과에 기반한 상태별 메시지 최적화

##### 시각적 구분 및 포맷팅
- 변경 유형에 따른 차별화된 알림 메시지 
- 상태 변화에 따른 이모지 사용으로 시각적 구분
  - 예약 가능해짐(✅), 매진됨(❌), 에러 발생(⚠️) 등
- 변경 유형 기반 알림 제목 자동 생성:
  - 매진→예약가능, 예약가능→매진, 에러발생, 에러해결 등
- 시간 정보 정렬 및 보기 좋은 포맷팅 기능
- 매수 정보 정렬 및 중요 정보 강조 표시 (3매 이상 굵게 표시)

##### 채널별 최적화
- 알림 채널별(텔레그램, 데스크톱) 최적화된 메시지 형식
- 긴 URL 자동 단축 처리로 가독성 향상
- 날짜 정보에 D-day 형식 추가 (D-3, D-day, D+1 등)
- 빠른 링크 기능으로 알림에서 바로 페이지 접속 가능
- 데스크톱 알림에 요약 정보 제공 (처음 3개 항목만 표시 + 추가 항목 수)

#### 8.2.3 중복 메시지 필터링 시스템
##### 중복 감지 메커니즘
- 최근 발송된 메시지 기록 유지 및 중복 메시지 자동 필터링
- 알림 메시지 해시 기반 중복 확인
- URL 별 상태 정보 기반 중복 필터링

##### 시간 기반 필터링
- 시간 기반 메시지 만료 처리 (설정 가능한 만료 시간)
- 시간별 알림 로그를 통한 동일 시간대 반복 알림 방지 (10분 이내)
- 시간대별 지능형 중복 필터링 적용

##### 시스템 구성
- 텔레그램 및 데스크톱 알림에 대한 중복 필터링 적용
- 메시지 중복 감지 로깅으로 시스템 동작 확인 용이
- 설정으로 활성화/비활성화 가능한 유연한 구조

#### 8.2.4 에러 페이지 감지 및 처리
##### 에러 페이지 감지
- URL 패턴 기반 에러 페이지 자동 감지 ("error", "invalidBusiness" 등)
- HTTP 상태 코드 기반 에러 감지 추가
- 자바스크립트 성능 API 활용 응답 상태 확인

##### 에러 유형별 처리
- 에러 유형별 차별화된 처리 로직:
  - invalidBusiness: 이미 매진된 상태로 간주하여 별도 알림 없음
  - 일반 오류: 디버깅 로그 및 후속 처리
- 예약 마감 상태 전용 처리 로직
- 접근 불가 페이지 특별 처리
- 변경 처리 시 유효성 검사 결과 활용
- 유효성 검사 결과에 따른 알림 메시지 차별화

##### 에러 관리 및 복구
- 연속 에러 발생 시 특별 관리자 알림 기능
- 에러 발생 시 대기 시간 자동 조정 (일반 대기 시간의 1/4로 단축)
- 에러 복구 후 자동 정상화 (에러 카운터 초기화)
- 설정 가능한 에러 패턴 및 처리 방식

### 8.3 시스템 최적화 및 유지보수
#### 8.3.1 브라우저 탭 관리 최적화

- 모니터링 대상별 탭 풀 관리로 리소스 효율화
- 설정 가능한 최대 탭 수 제한(MAX_TABS_PER_WORKER)으로 시스템 안정성 확보
- 일정 시간(기본값 50분) 사용되지 않은 탭 자동 정리
- 강제 정리 모드에서 최소 하나의 탭 유지 기능

##### 안전한 탭 사용 시스템
- 안전한 탭 사용 상태 추적 시스템:
  - 각 탭의 사용 중(in-use) 상태 명시적 추적
  - 작업 종료 보장을 위한 탭 반환(release) 메커니즘
  - try-finally 구조를 통한 확실한 탭 반환 보장
  - 탭 ID 및 대상 ID 정보 저장으로 작업 완료 시 정확한 참조

##### 부하 분산 및 탐지 방지
- 랜덤 탭 선택 알고리즘을 통한 탐지 방지 및 부하 분산:
  - 자유롭게 사용 가능한(non-in-use) 탭만 선택 대상으로 고려
  - 소수 탭 상황에서는 최근 사용 탭 재사용 (캐시 활용)
  - 다수 탭 상황에서는 랜덤 선택으로 탐지 방지
- 모든 탭이 사용 중인 경우 대기-재시도 메커니즘:
  - 가용 탭이 없는 경우 짧은 대기 후 재시도
  - 재귀적 호출을 통한 안정적인 탭 획득 보장
- 탭 생성 시 무작위 ID 부여로 추적 어려움 증가

##### 메모리 관리 최적화
- 탭 캐싱 및 재사용으로 불필요한 브라우저 인스턴스 생성 최소화
- 강제 탭 회전:
  - 설정 가능한 요청 횟수마다 강제 탭 교체로 메모리 누수 방지
  - 설정 가능한 요청 횟수마다 캐시 및 스토리지 정리 수행
- 페이지 초기화 절차:
  - 재사용 탭에 about:blank 로드로 기존 페이지 리소스 정리
  - 세션 및 로컬 스토리지 정리로 메모리 누수 방지

##### 상태 관리 및 복구

- 장애 복구 시 탭 풀 자동 정리 및 초기화
- 설정 파일 기반 탭 관리 파라미터 최적화
